A microkernel where authority is held, never assumed.

Horus treats the capability token as its single foundational security primitive. Every privileged operation — file access, IPC, task creation, signalling, device I/O — requires an explicit, unforgeable capability. There is no ambient authority to fall back on: a task with no capability for an object cannot even name it.

Overview

A C kernel with a security core the compiler can check.

The kernel is written in C. The security-critical core — the capability engine, physical-memory reference counting, the cryptographic primitives, the W^X page policy, and every FFI validation boundary — is implemented in safe, no_std Rust, where the type system statically rules out entire classes of memory-safety defects. The C/Rust ABI is pinned by mirrored compile-time assertions on both sides of the boundary.

Capabilities can be minted with reduced rights, delegated between tasks, and revoked instantly and transitively across the whole system. A spawned task receives only the TCB, frame, and endpoints it needs — never the admin, block-device, or console capabilities. A parent delegates one slot at a time into a child it supervises.

Horus is engineered as if it were destined for production even though it is not. Every change is gated by CI: the unit-test suite, a linter with all warnings denied, a byte-for-byte reproducible-build check, a suite of headless QEMU self-tests, and a supply-chain scan that emits an SBOM.

Principles

Five commitments, applied without exception.

These are not aspirations written after the fact — each one is enforced somewhere you can go read.

No ambient authority

Access derives solely from held capabilities — never from UID, task identity, or global state. Authority is a thing you were handed, not a thing you are.

Least privilege by construction

Capabilities are minted with a subset of rights. Privilege narrows as it is passed down, and it is structurally awkward to widen it — the easy path is the safe one.

Verifiable core

Security-sensitive logic lives in safe Rust with unit tests and known-answer cryptographic vectors, checked against published RFC test vectors.

Defence in depth

SMEP/SMAP and W^X/NX, a single centralised syscall-authorisation choke point, transitive revocation, and a tamper-evident audit log reinforce one another.

Provenance you can trust

Reproducible builds and a first-party-only CI supply chain mean a released kernel.elf can be independently reproduced bit-for-bit.

Verify

Don't take the table's word for it.

Every claim above is a command you can run. x86-64 host, Debian or Ubuntu. 

# build and boot
$ make              # kernel.elf
$ make run          # boot.iso + QEMU
                    # console on serial:
                    # nc localhost 4445
# prove it
$ make test              # 58 Rust unit tests
$ make smoke             # boot to ring-3 login
$ make smoke-proc        # process control
$ make reproducible-build # bit-for-bit

The default login is user / password. Full toolchain reference, every target, and troubleshooting live in BUILDING.md.

Status

Where the line sits.

Research / early development

Horus boots, runs a ring-3 init that supervises a ring-3 shell, and enforces capability-based access control end to end. It has preemptive scheduling, a userspace filesystem server over an encrypted object store, a newlib libc port, ring-3 process control, and multi-core support behind a build gate.

Some subsystems — SMP default-on, multi-slot IPC — are deliberately scaffolded rather than finished. This is a research and learning kernel, not a shipping OS. It has not been audited, and nothing here should be trusted with anything real.

LIMITATIONS.md is a candid, subsystem-by-subsystem account of exactly where that line sits — written to be read before the README's feature list, not after.
